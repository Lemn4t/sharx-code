## Release v1.6.5

**Release Date:** June 18, 2026

Патч **v1.6.5** относительно **v1.6.4**: IP-limit job больше не перезапускает все ноды при проверке онлайна.

**Миграция БД:** нет.

---

### Bug Fixes

- **Рестарт всех нод из-за IP-limit / session block** (`web/service/xray.go`, `web/service/client_session_block.go`, `xray/session_ip_block_routing.go`)
  - **Симптом:** в логах ноды `session-ip-block-routing: proto: cannot parse invalid wire-format data`, сразу за ним `apply-config` → `compare result=changed` → рестарт Xray при тех же инбаундах.
  - **Причина 1:** job IP-limit (`CheckClientIPLimitJob`) на каждом цикле снова вызывал hot-push для **уже заблокированных** IP.
  - **Причина 2:** при ошибке hot routing панель делала **fallback на полный `apply-config`** на все упавшие ноды — routing rules в payload отличались от running core → лишний restart storm.
  - **Фикс:** не повторять hot-push для IP, уже в `client_blocked_session_ips`; убран `apply-config` fallback; сборка routing rule для `RoutingService.AddRule` через `infra/conf.ToCidrList` (тот же путь, что JSON routing в конфиге).

### Operational Notes

- Обновляется **панель (`sharx`)**; **`sharxnode`** желательно обновить вместе (общий `xray` пакет для hot routing API).
- После деплоя: при включённом **IP limit** ноды не должны рестартовать каждые N секунд; в логах панели — warning вместо каскада `apply-config`.
- Если hot routing на старой ноде всё ещё падает (нет `RoutingService` в api.services) — блокировка IP вступит при следующем штатном sync конфига, без рестарта по кругу.

### Test plan

- [ ] Settings → IP limit включён, клиент с лишними online IP → block enforcement срабатывает
- [ ] Логи ноды: нет повторяющихся `apply-config` / `action=restart` каждые 30s при стабильном конфиге
- [ ] Ручной block/unblock IP в Clients → sessions → hot routing или warning, без массового рестарта
